# Internationalization (i18n)

Pinmok follows Django's i18n system without introducing any additional configuration layer. `padmin` ships with built-in translations for
Simplified Chinese (`zh-hans`) and English (`en`). Developers add translations to their own apps using the standard Django workflow.

In addition to standard i18n support, Pinmok provides two tools:

- **Translatable models** (`TranslatableModel` / `TranslationModel`): for database-level multilingual content using a main table +
  translation table pattern
- **PO deduplication tool** (`dedupe_po`): cleans duplicate entries from `.po` files

---

## Configuration

Enable standard Django i18n settings in `settings.py`:

```python
USE_I18N = True
LANGUAGE_CODE = 'zh-hans'  # default language

LANGUAGES = [
    ('zh-hans', 'Simplified Chinese'),
    ('en', 'English'),
]
```

Add `LocaleMiddleware` to `MIDDLEWARE`, positioned after `SessionMiddleware` and before `CommonMiddleware`:

```python
MIDDLEWARE = [
    ...
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.locale.LocaleMiddleware',  # ← here
    'django.middleware.common.CommonMiddleware',
    ...
]
```

To switch languages via URL prefix (e.g., `/zh-hans/`, `/en/`), use `i18n_patterns` in `urls.py`:

```python
from django.conf.urls.i18n import i18n_patterns

urlpatterns += i18n_patterns(
    path('', include('your_app.urls')),
)
```

The `padmin` interface language follows Django's active language environment — no additional configuration required.

## Translatable Models

**Use case:** content data that requires multilingual versions, such as category names, article titles, or navigation labels. This pattern
uses a **main table + translation table** structure: the main table stores language-agnostic fields, and the translation table stores the
localized text for each language.

### Defining Models

The main model inherits from `TranslatableModel`; the translation model inherits from `TranslationModel`:

```python
from pinmok.core.models import TranslatableModel, TranslationModel
from pinmok.core.constants import TRANSLATION_RELATED_NAME
from django.db import models


class Category(TranslatableModel):
    sort_order = models.IntegerField(default=0)
    is_active = models.BooleanField(default=True)

    class Meta:
        ordering = ['sort_order']


class CategoryTranslation(TranslationModel):
    category = models.ForeignKey(
        Category,
        on_delete=models.CASCADE,
        related_name=TRANSLATION_RELATED_NAME,  # must be 'translations'
    )
    name = models.CharField(max_length=255)
    description = models.CharField(max_length=200, blank=True, default='')

    class Meta(TranslationModel.Meta):
        constraints = [
            models.UniqueConstraint(
                fields=['category', 'language'],
                name='uniq_category_language'
            )
        ]

    def get_display_text(self):  # must be implemented
        return self.name
```

**Notes:**

- The FK `related_name` on the translation model must be `'translations'`. Use the constant `TRANSLATION_RELATED_NAME` rather than
  hardcoding the string.
- `UniqueConstraint` must be defined to prevent duplicate entries for the same language.
- `get_display_text()` must be implemented — the framework's `__str__` and `translation` property both depend on it.

### What the Framework Handles Automatically

Once you inherit from these base classes, the following works out of the box:

**Get the translation for the current language:**

```python
category = Category.objects.get(pk=1)
category.translation.name  # returns current language, with fallback
```

**Fallback priority:** exact language match → base language code (`zh-hans` → `zh`) → `LANGUAGE_CODE` → first available translation.

**Get a specific language:**

```python
category.get_translation('en').name
```

**`__str__` proxies to the translation:**

```python
str(category)  # returns get_display_text() in the current language
```

### What You Still Need to Implement

- Define the translated fields on the translation model (`name`, `description`, etc.)
- Implement `get_display_text()`
- Register the translation model in `admin.py`, typically as a `StackedInline` or `TabularInline`:

```python
from django.contrib import admin
from .models import Category, CategoryTranslation


class CategoryTranslationInline(admin.StackedInline):
    model = CategoryTranslation
    extra = 1


@admin.register(Category)
class CategoryAdmin(admin.ModelAdmin):
    inlines = [CategoryTranslationInline]
```

### Avoiding N+1 Queries

In list views or bulk processing, use `with_translations()` to prefetch translation data:

```python
# in a view or serializer
categories = Category.with_translations()

# also works on an existing queryset
categories = Category.with_translations(
    Category.objects.filter(is_active=True)
)
```

Once prefetched, the `translation` property and `get_translation()` do not trigger additional queries.

### Refreshing the Cache After Saving

If you save a translation record and need to read the updated value within the same request, manually clear the instance cache:

```python
translation.save()
category.invalidate_translation_cache()
```

## PO Deduplication Tool

Django's `makemessages` can sometimes produce duplicate entries in `.po` files. The `dedupe_po` tool removes these duplicates and can also
strip entries that duplicate Django's own built-in translations — strings already covered by `django.contrib.admin`, `django.contrib.auth`,
and similar packages do not need to be maintained in your app's `.po` files.

### Usage

Run from the project root:

```bash
# process default language (zh-hans), all domains
python manage.py dedupe_po

# specify a language
python manage.py dedupe_po -l en

# django domain only
python manage.py dedupe_po -d django

# specify the scan directory
python manage.py dedupe_po --dir /path/to/project
```

The tool recursively scans `locale/<lang>/LC_MESSAGES/` under all app directories within the project. All apps are processed in a single
run — no need to handle them one by one.

### Options

| Option          | Default           | Description                                                |
|-----------------|-------------------|------------------------------------------------------------|
| `-l, --lang`    | `zh-hans`         | Target language code                                       |
| `-d, --domain`  | all               | Restrict to a specific domain: `django` or `djangojs`      |
| `--dir`         | current directory | Root directory to scan                                     |
| `--no-backup`   | —                 | Skip backup creation (a `.bak` file is created by default) |
| `--no-comments` | —                 | Do not preserve comments                                   |

### Backups

By default, a timestamped backup is created in the same directory before each file is modified:

```
django_250601120000.bak
```

Delete backup files manually once you have confirmed the results. Pass `--no-backup` to skip this step.
