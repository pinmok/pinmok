# Built-in Template Tags

Pinmok provides a set of built-in template tags and filters for direct use in templates. These are organized into two files by scope:
back-end-only tags live in `pinmok_admin_tags`, while tags shared between front-end and back-end live in `pinmok_tags`.

## Back-end Tags

Tags designed exclusively for back-end template rendering.

Load the tag library in templates:

```django
{% load pinmok_admin_tags %}
```

### add_class

**Filter**

Django Admin form fields are rendered by widgets that output complete HTML strings, leaving no direct way to modify element attributes at
the template level. The `add_class` filter operates on rendered HTML strings to append CSS class names to field elements without rewriting
widget logic — well suited for lightweight style adjustments.

**Usage:**

```django
{{ field.label_tag|add_class:"form-label required" }}
{{ field.field|add_class:"form-control is-invalid" }}
```

Works with any HTML element. If the element already has a `class` attribute, the new class names are appended at the end; if not, a `class`
attribute is created automatically. Each call affects only the first tag in the string.

### alert

**Inclusion Tag**

Wraps Tabler icon components and standard HTML structure to render alert components quickly, without writing style classes and icon markup
by hand.

**Usage:**

```django
{% alert "Operation successful" level="success" %}
{% alert "Please note" level="warning" description="This action cannot be undone." dismiss=True %}
{% alert "An error occurred" level="danger" link_url="/help/" link_text="View details" %}
```

**Parameters:**

| Parameter     | Type | Default    | Description                                                          |
|---------------|------|------------|----------------------------------------------------------------------|
| `title`       | str  | Required   | Alert title; nothing is rendered if empty                            |
| `level`       | str  | `'danger'` | Alert type: `danger` / `warning` / `success` / `info`                |
| `description` | str  | `None`     | Optional body text displayed below the title                         |
| `variant`     | str  | `''`       | Visual variant: `important` / `minor`; leave blank for default style |
| `dismiss`     | bool | `False`    | Renders a close button when `True`                                   |
| `extra_class` | str  | `''`       | Additional CSS classes appended to the alert element                 |
| `link_url`    | str  | `None`     | Action link URL; requires `link_text` to take effect                 |
| `link_text`   | str  | `None`     | Action link text; requires `link_url` to take effect                 |

## General Tags

Tags and filters available in both front-end and back-end templates.

Load the tag library in templates:

```django
{% load pinmok_tags %}
```

### truncate_filename

**Filter**

Truncates long file names while preserving the file extension, preventing overflow when displaying file names in the UI.

**Usage:**

```django
{{ filename|truncate_filename }}
{{ filename|truncate_filename:30 }}
```

The default maximum length is 20 characters. When exceeded, the base name is truncated and replaced with an ellipsis (`…`); the extension is
always preserved.

### icon

**Simple Tag**

Renders an inline SVG icon from a sprite file. Icon names prefixed with `tabler-` use Pinmok's built-in sprite file; all other names
reference a custom sprite file whose default path can be overridden in `settings.py` via `CUSTOM_SPRITE_FILE`.

> All available icons can be browsed in the back-end **Icon Manager**.

**Usage:**

```django
{% icon "tabler-home" %}
{% icon "tabler-user" "icon nav-icon" %}
{% icon "my-custom-icon" "icon" 20 %}
```

**Parameters:**

| Parameter   | Type | Default  | Description                                 |
|-------------|------|----------|---------------------------------------------|
| `icon_name` | str  | Required | Symbol ID in the sprite file                |
| `css_class` | str  | `''`     | CSS classes appended to the `<svg>` element |
| `size`      | int  | `24`     | SVG width and height in pixels              |

### media_url

**Simple Tag**

Resolves a media file path to a full URL.

Django's `FieldFile` exposes a `.url` property that can be used directly in templates. However, raw path strings retrieved from the database
or cache do not carry this property — `media_url` is designed for exactly that situation.

It resolves paths through Django's `default_storage` backend, so its behavior stays consistent with whatever storage backend the project
uses:

- With local storage, it returns the path joined with `MEDIA_URL`.
- With a third-party backend (such as S3 or Alibaba Cloud OSS) or a CDN, it returns the full URL generated by that backend.

As long as the storage backend is correctly configured, `media_url` produces an accessible URL in any environment — no manual path
concatenation required.

**Usage:**

```django
{# Raw path string #}
{% media_url object.image as url %}
<img src="{{ url }}">

{# Used with site_info #}
{% site_info as site %}
{% media_url site.site_logo as logo_url %}
<img src="{{ logo_url }}">
```

Absolute URLs (starting with `http://` or `https://`) are returned as-is. An empty or unresolvable path returns an empty string.

### site_info

**Simple Tag**

Returns a site configuration dictionary sourced from the Site Information category in the back-end site settings.

**Usage:**

```django
{% site_info as site %}
<title>{{ site.name }}</title>
```

**Available fields:**

| Variable                       | Description                                |
|--------------------------------|--------------------------------------------|
| `site.site_name`               | Site name                                  |
| `site.site_slogan`             | Site slogan                                |
| `site.site_logo`               | Site logo image path                       |
| `site.icp`                     | ICP filing number                          |
| `site.pns`                     | Public security network filing number      |
| `site.service_phone`           | Service phone number                       |
| `site.service_email`           | Service email address                      |
| `site.contact_address`         | Contact address                            |
| `site.wechat_qrcode`           | WeChat QR code image path                  |
| `site.wechat_mini_program`     | WeChat Mini Program QR code image path     |
| `site.wechat_official_account` | WeChat Official Account QR code image path |
| `site.facebook_link`           | Facebook link                              |
| `site.x_link`                  | X link                                     |
| `site.linkedin_link`           | LinkedIn link                              |
| `site.instagram_link`          | Instagram link                             |

Image fields return storage paths. Use them together with `{% media_url %}` to generate accessible URLs.

### links

**Simple Tag**

Returns the list of enabled external links sorted by `sort_order`. Results are cached to reduce database queries. Each entry contains
`title`, `url`, and `image` fields.

**Usage:**

```django
{% links as link_list %}
{% for link in link_list %}
    <a href="{{ link.url }}">{{ link.title }}</a>
{% endfor %}
```

### Navigation Tags

**Block Tag**

Renders a navigation tree from a given group, supporting arbitrary nesting depth. `navigation` handles data loading and recursive rendering;
`navblock` defines the HTML fragment for each level. The two tags must be used together.

Navigation data is managed in the back end and organized by group to distinguish different navigation positions (main navigation, footer,
sidebar, etc.). Group names come from template configuration variables selected in the back end — see the Template Configuration chapter for
details.

#### Basic Structure

```django
{% navigation nav.category %}
    {% navblock 1 %}
        <!-- HTML for the first level -->
    {% endnavblock %}
    {% navblock 2 %}
        <!-- HTML for the second level -->
    {% endnavblock %}
    {% navblock default %}
        <!-- HTML for all remaining levels -->
    {% endnavblock %}
{% endnavigation %}
```

#### Level Matching Rules

When rendering a level, the tag first looks for a `navblock` matching that level number. If none is found, it falls back to `default`; if
neither exists, that level is not rendered. In most cases, defining `navblock 1` and `navblock default` is enough to cover all levels.

#### Variables Available Inside `navblock`

| Variable        | Type | Description                                                                                             |
|-----------------|------|---------------------------------------------------------------------------------------------------------|
| `item.url`      | str  | Navigation item link; parent groups may have no actual URL — use `item.url\|default:'#'`                |
| `item.name`     | str  | Navigation item display name                                                                            |
| `item.icon`     | str  | Icon name for use with `{% icon %}`; empty string when no icon is set — check before using              |
| `item.target`   | str  | Link `target` attribute, e.g. `_blank`; empty string when not set                                       |
| `item.children` | list | Child node list; useful for checking whether the current item has children                              |
| `children`      | str  | Recursively rendered HTML of child nodes; output directly — no need to iterate `item.children` manually |

#### Complete Example

The following example implements two-level navigation: first-level items render as navbar entries and become dropdowns when they have
children; second-level items and below render as dropdown items and support nested dropdowns.

```django
{% navigation nav.category %}

    {% navblock 1 %}
        {% if item.children %}
            <li class="nav-item dropdown">
                <a class="nav-link dropdown-toggle"
                   href="{{ item.url|default:'#' }}"
                   data-bs-toggle="dropdown"
                   role="button">
                    {% if item.icon %}
                        <span class="nav-link-icon">{% icon item.icon 'icon' %}</span>
                    {% endif %}
                    <span class="nav-link-title">{{ item.name }}</span>
                </a>
                <div class="dropdown-menu">{{ children }}</div>
            </li>
        {% else %}
            <li class="nav-item">
                <a class="nav-link" href="{{ item.url }}">
                    {% if item.icon %}
                        <span class="nav-link-icon">{% icon item.icon 'icon' %}</span>
                    {% endif %}
                    <span class="nav-link-title">{{ item.name }}</span>
                </a>
            </li>
        {% endif %}
    {% endnavblock %}

    {% navblock default %}
        {% if item.children %}
            <div class="dropend">
                <a class="dropdown-item dropdown-toggle"
                   href="{{ item.url|default:'#' }}"
                   data-bs-toggle="dropdown"
                   role="button">
                    {{ item.name }}
                </a>
                <div class="dropdown-menu">{{ children }}</div>
            </div>
        {% else %}
            <a class="dropdown-item" href="{{ item.url }}">{{ item.name }}</a>
        {% endif %}
    {% endnavblock %}

{% endnavigation %}
```

`children` is the result of recursive rendering — when `navblock default` renders the second level, the `{{ children }}` inside it renders
the third level using the same fragment, and so on until no deeper items remain.

### slider

**Simple Tag**

Returns the list of enabled slides under the specified group, sorted by `sort_order`. Results are cached to reduce database queries.

Slider data is managed in the back end and organized by group to distinguish sliders at different page positions. Group names come from
template configuration variables selected in the back end — see the Template Configuration chapter for details.

#### Usage:

```django
{% slider 'home' as slides %}
{% for slide in slides %}
    <a href="{{ slide.link }}">
        <img src="{% media_url slide.image %}" alt="{{ slide.title }}">
        {% if slide.subtitle %}<p>{{ slide.subtitle }}</p>{% endif %}
    </a>
{% endfor %}
```

#### Returned Fields:

| Field      | Description                                                |
|------------|------------------------------------------------------------|
| `title`    | Slide title                                                |
| `subtitle` | Subtitle; may be empty                                     |
| `image`    | Image path; use `{% media_url %}` to resolve to a full URL |
| `link`     | Click-through link; may be empty                           |
